1. Secure Development
✓
Must Do
  • Follow secure coding standards — input validation, output encoding, proper authentication, authorization, and error handling
  • Ensure all production-bound changes have a peer review before merging
  • Use only approved languages, frameworks, and libraries; keep them up to date
✕
Don't
  • Don't hardcode secrets (passwords, tokens, keys, connection strings) in code, config files, or tests
  • Don't paste unvetted code from the internet into production systems without careful review
Never commit .env files to Git — use .gitignore and pre-commit hooks.
Practical Example

In a GitHub PR review, check if the developer added input validation to a new API endpoint before approving. If you spot a hardcoded API key, request changes immediately.

Best Practice
  • Store all secrets in the approved secret manager; inject via environment variables or secure configuration mechanisms
  • Use online code snippets as inspiration only — review every line for security, licensing, and compatibility before merging
  • Every production-bound change must pass peer code review — it is your strongest security checkpoint
2. Environment & Access Management
✓
Must Do
  • Maintain strict separation between dev, test, stage, and production
  • Restrict production access to those who strictly need it; protect with SSO + MFA
  • Reference tickets / change IDs when accessing or modifying production systems
✕
Don't
  • Don't make ad-hoc changes in production (DB updates, config edits, infra tweaks) outside the approved change process
  • Don't use raw production data in non-production environments without masking or explicit approval
Mandatory: SSO + MFA for all Production logins.
Practical Example

Your local 'localhost' DB password should never be the same as the Production RDS password. Moving to a new project? Send a ticket to IT to revoke your old project's database access.

Best Practice
  • Capture all changes in a ticket; implement via CI/CD or documented procedures with approvals in place
  • Use synthetic or masked datasets for development and testing — never real customer data
  • For a production data sample, work with DevOps/Manager to get an approved, anonymized export
3. Change & Release Management
✓
Must Do
  • Raise a change ticket for all production changes — include description, risk, approvals, and rollback plan
  • Use CI/CD pipelines to build, test, and deploy wherever possible
  • Validate deployments with smoke tests, log checks, and monitoring dashboards
✕
Don't
  • Don't bypass pipeline checks (tests, linters, security scans) without a documented, time-bound exception
  • Don't push "quick fixes" directly to production without traceability, testing, or a ticket reference
Practical Example

Before running a DB migration, document the exact SQL command to 'undo' the change if the deployment fails. Every hotfix must still go through the pipeline with a ticket reference.

Best Practice
  • If a pipeline check is blocking for a valid reason — document the rationale, get explicit approval, and schedule the fix as soon as possible
  • Implement all fixes as small, well-scoped changes through the normal pipeline — with a ticket reference and a tested rollback plan
  • No ticket = no traceability. Every production change must be tracked.
4. Secrets, Encryption & Configuration
✓
Must Do
  • Store all secrets in approved secret managers (cloud KMS / Secrets Manager, Vault)
  • Use HTTPS / TLS for all external and sensitive internal communication
  • Enable encryption at rest for databases, object storage, and volumes as required by policy
✕
Don't
  • Don't commit secrets to git or include them in tickets, chat messages, or logs
  • Don't leave default credentials or demo accounts active in any environment after deployment or POC
Never commit .env files to Git or share secrets in Slack/Teams messages.
Practical Example

Store your Stripe API Key in AWS Secrets Manager and fetch it at runtime using an IAM role. If a secret is accidentally committed, rotate it immediately — don't just delete the commit.

Best Practice
  • If a secret is accidentally exposed — rotate it immediately, remove it from git history where possible, and document the incident
  • After any deployment or POC — change all default credentials and remove demo accounts; only required named accounts should remain
  • Treat exposed secrets as security incidents — always report them to compliance@mindbowser.com
5. Logging, Monitoring & Observability
✓
Must Do
  • Log key events: authentication attempts, permission changes, critical actions, and major errors
  • Use structured logging formats; never log PHI, passwords, tokens, or secrets
  • Work with Security/Ops to configure alerts for critical error and security patterns
✕
Don't
  • Don't disable or reduce logging/monitoring in production to "save time"
  • Don't delete or alter logs related to incidents — even if they are verbose or costly
Don't log full request bodies containing PII or PHI.
Practical Example

Log { "event": "user_login", "status": "failed", "user_id": 123 } to CloudWatch or ELK. If a user updates their profile, log "fields_updated": ["email", "address"] but not the actual values.

Best Practice
  • Tune logging levels and filters to manage noise — but always preserve security-relevant events and traceability
  • If logs are too verbose or costly, update logging strategy going forward — never delete historical logs
  • Think of logs as CCTV footage — the moment you delete them, you lose your evidence
6. Vulnerability & Risk Management
✓
Must Do
  • Fix vulnerabilities within defined SLAs — especially critical and high severity issues
  • Address findings from SAST, DAST, dependency scans, and pen tests — or document risk acceptance with formal approvals
  • Participate in design reviews and threat modeling for new services and major architecture changes
✕
Don't
  • Don't ignore vulnerability notifications from scanners, researchers, customers, or vendors
  • Don't deploy known-vulnerable components into production without an approved exception and compensating controls
Practical Example

If Snyk or GitHub Advanced Security flags a SQL Injection risk, prioritize the fix in the current sprint. Never deploy known-vulnerable packages without documented approval and compensating controls (e.g., WAF rules).

Best Practice
  • Triage each finding — determine impact and priority, then either schedule remediation or go through a formal risk acceptance process
  • If deploying with known risk — obtain documented approval, add compensating controls (e.g., WAF rules), and track remediation
  • Report all security findings through compliance@mindbowser.com
7. Data Protection & Privacy in Applications
✓
Must Do
  • Collect and store only the minimum data required to deliver the feature
  • Classify data fields (PHI, personal data) and apply stronger technical and process controls accordingly
  • Implement data retention and deletion rules in applications — cleanup jobs, lifecycle policies
✕
Don't
  • Don't log sensitive fields such as passwords, tokens, PHI, or personal identifiers
  • Don't expose internal details or sensitive data via debug endpoints or verbose error responses in production
Follow the 'Clean Desk & Clean Drive' policy — delete or archive data per retention schedule.
Practical Example

Set an S3 Lifecycle Policy to move logs older than 1 year to Glacier or delete them. A list of team birthdays is 'Internal'; a patient's medical history is 'Confidential/Restricted'.

Best Practice
  • Log non-sensitive identifiers (e.g., internal IDs) and generalized error messages — never raw sensitive data
  • Use generic error messages for users; keep detailed diagnostics in internal logs accessible only to authorized staff
  • Data you don't collect can never be leaked — minimize collection from day one
8. AI & Automation in Engineering
✓
Must Do
  • Use only company-approved AI tools for code suggestions, refactoring, and documentation
  • Treat AI-generated code and configs as drafts — review thoroughly for security, correctness, and licensing before merging
  • Use generic or masked examples in prompts; never real customer identifiers or proprietary algorithms
✕
Don't
  • Don't paste full repositories, production configs, customer data, or secrets into public AI tools
  • Don't auto-merge AI-generated changes without human review, tests, and approvals
Check the internal AI Usage Policy before using any AI tool for work.
Practical Example

Instead of asking AI: "Find the error in this patient record for Jane Doe...", say: "Find the logic error in this JSON object with a string field..." Use ChatGPT for brainstorming, never for processing real customer data.

Best Practice
  • Share small, abstracted snippets with sensitive details removed when asking AI for help — or use internal AI solutions explicitly approved for such use
  • Run all AI-suggested changes through the normal code review, testing, and approval process before merging
  • Treat AI like a junior developer — it gives ideas, but you own the final output and its consequences
What SOC 2 Means for Engineers
Every Engineer at Mindbowser must:
  • Never hardcode secrets or commit them to version control
  • Keep production access restricted — always enforce SSO + MFA
  • Raise a ticket before every production change — no exceptions
  • Use CI/CD and peer review for every deployment
  • Log security events and never delete or alter logs
  • Fix vulnerabilities within defined SLA; never ignore scanner findings
  • Use AI responsibly — approved tools only, review all output, no secrets in public AI tools
