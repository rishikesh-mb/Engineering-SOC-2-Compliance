Area 01

Secure Development

✓
Must Do
  • Follow secure coding standards (input validation, auth, error handling).
  • Ensure all production-bound changes have a peer review before merging.
  • Use only approved languages, frameworks, and libraries; keep them updated.
Shift-Left Mentality
  • Catch vulnerabilities directly in the IDE or PR phase. Security is a feature, not an afterthought.
✕
Don't Do
  • Do not hard-code secrets (passwords, tokens, keys) in code, config, or tests.
  • Do not paste unvetted code directly from the internet into production without careful review.
Instead

Store secrets in the approved manager and inject via Env Vars. Use online snippets as inspiration, but run them through normal code review.

Area 02

Environment & Access Management

✓
Must Do
  • Maintain strict separation between dev, test, stage, and prod environments.
  • Restrict production access to those who strictly need it (SSO + MFA).
  • Reference tickets / change IDs when modifying production.
Defense in Depth
  • Every layer (network, application, database) must verify identity independently.
✕
Don't Do
  • Do not make ad-hoc changes in production (DB updates, config edits) outside the process.
  • Do not use raw production data in non-production environments without masking.
Instead

Capture the need in a ticket and implement via CI/CD. Use synthetic datasets for development or request an approved, anonymized export.

Area 03

Change & Release Management

✓
Must Do
  • Raise a change ticket for all prod changes (incl. scope, risk, rollback plan).
  • Use CI/CD pipelines to build, test, and deploy where possible.
  • Validate deployments with smoke tests and log checks.
✕
Don't Do
  • Do not bypass pipeline checks (tests, security scans) without a documented exception.
  • Do not push "quick fixes" directly to production without testing.
Instead

If a check blocks a valid deployment, get explicit approval and document it. Implement hotfixes as well-scoped changes via the pipeline with a ticket.

Area 04

Secrets, Encryption & Configuration

✓
Must Do
  • Store all secrets in approved secret managers (AWS KMS, Vault).
  • Use HTTPS / TLS for all external and sensitive internal comms.
  • Enable encryption at rest for DBs and object storage.
✕
Don't Do
  • Do not commit secrets to git, tickets, chat, or logs.
  • Do not leave default credentials or demo accounts active in prod.
Instead

If accidentally exposed, rotate it immediately, remove from git history, and report it. Delete demo accounts post-POC.

Area 05

Logging, Monitoring & Observability

✓
Must Do
  • Log key security/business events: auth attempts, permission changes, major errors.
  • Use structured logging formats where possible.
  • Work with Ops to configure alerts for security patterns.
✕
Don't Do
  • Do not log PHI, passwords, tokens, or full request bodies.
  • Do not disable/delete logs related to incidents to "save time/space".
Instead

Tune log filters moving forward if too noisy, but preserve historical logs for investigations. Log "fields_updated": ["email"] instead of raw values.

Area 06

Vulnerability & Risk Management

✓
Must Do
  • Fix vulnerabilities within defined SLAs (Critical/High priority).
  • Address findings from SAST, DAST, and dependency scans.
  • Participate in threat modeling for major architecture changes.
✕
Don't Do
  • Do not ignore notifications from scanners or researchers.
  • Do not deploy known-vulnerable components without exception controls.
Instead

Triage each finding. If you must deploy with a known risk, obtain documented approval and apply compensating controls (e.g., WAF rules).

Area 07

Data Protection & Privacy in Apps

✓
Must Do
  • Collect and store only the minimum data required.
  • Classify data fields (e.g., PHI, internal) and apply appropriate technical controls.
  • Implement data retention and deletion rules (cleanup jobs).
✕
Don't Do
  • Do not log sensitive fields or PII.
  • Do not expose internal details via verbose error responses in production.
Instead

Use generic error messages for end-users. Set an S3 Lifecycle Policy to delete logs older than 1 year.

Area 08 & Part C

AI & Automation / Responsible Use

✓
Must Do
  • Use company-approved AI tools for drafting/refactoring.
  • Treat AI code as drafts requiring thorough human review.
  • Use generic/abstracted examples in prompts.
✕
Don't Do
  • Do not paste full repos, configs, or customer data into public AI.
  • Do not auto-merge AI changes without tests.
  • Do not rely solely on AI for security or compliance decisions.
Instead

If debugging, say "Find the logic error in this JSON object" rather than pasting the actual user object. You own the final output, not the AI.
