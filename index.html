SECURITY MODULE 2025

Deep Dive:
Security Standards

A granular guide to SOC 2 Compliance and Data Safety at Mindbowser Inc.

01 Identity: SSO

  • Use Single Sign-On for all company tools.
  • Centralized access prevents rogue accounts.
Practical Example

Always click "Login with Google/Okta" using your @mindbowser.com email instead of creating a separate password for tools like Jira or HROne.

02 Identity: MFA

  • Multi-Factor Authentication is mandatory.
  • Protects against compromised passwords.
Never share your MFA code/OTP with anyone.
Practical Example

If you receive an unexpected Duo/Google Authenticator prompt while not logging in, click 'Deny' and report it immediately.

03 Access Management

  • Request only the access you need for your role.
  • Revoke access immediately when rolling off.
Practical Example

Moving from Project A to Project B? Send a ticket to IT to remove your Project A database permissions.

04 Endpoint Security

  • Company devices must have Antivirus enabled.
  • Software auto-updates must never be blocked.
Compliance check: Ensure Disk Encryption is ON.
Practical Example

If your Mac prompts for a 'Security Update', install it during your lunch break rather than clicking 'Remind me tomorrow'.

05 Physical Security

  • Auto-screen lock is set to 15 minutes.
  • Manually lock when leaving your desk.
Practical Example

Walking to the office kitchen for coffee? Press 'Win+L' or 'Cmd+Ctrl+Q' before standing up.

06 Secure Dev: PRs

  • No code reaches production without peer review.
  • Reviewers must check for logic & security.
Practical Example

In a GitHub PR review, check if the developer added input validation to a new API endpoint before approving.

07 Secrets Management

  • No hard-coded secrets in codebases.
  • Use Secret Managers (Vault/AWS Secrets).
Never commit .env files to Git.
Practical Example

Store your Stripe API Key in AWS Secrets Manager and fetch it at runtime using an IAM role.

08 Env Isolation

  • Dev, Stage, and Prod must be strictly isolated.
  • Credentials must differ across environments.
Practical Example

Your local 'localhost' DB password should never be the same as the Production RDS password.

09 Production Access

  • Access granted only on 'Need to Know' basis.
  • All Prod access requires a valid Ticket ID.
Mandatory: SSO + MFA for all Prod logins.
Practical Example

Logging into the AWS Console? Enter the Jira Ticket ID (e.g., MB-102) in the 'Reason' field of your access request.

10 Change Management

  • All Prod changes must be documented.
  • Include Scope, Risk, and Rollback plan.
Practical Example

Before running a DB migration, document the exact SQL command to 'undo' the change if the deployment fails.

11 Logging & Monitoring

  • Log auth attempts, permission changes, and errors.
  • Use structured formats (JSON) for searching.
Practical Example

Log { "event": "user_login", "status": "failed", "user_id": 123 } to CloudWatch or ELK.

12 Logging: Privacy

  • Never log PHI, passwords, or secrets.
  • Scrub sensitive fields at the application level.
Don't log full request bodies containing PII.
Practical Example

If a user updates their profile, log "fields_updated": ["email", "address"] but not the actual values.

13 Vulnerability Patching

  • Fix High/Critical issues within 30 days.
  • Participate in regular Pen Tests.
Practical Example

If Snyk or GitHub Advanced Security flags a SQL Injection risk, prioritize the fix in the current sprint.

14 Data Classification

  • Internal: Default company info.
  • Confidential: Customer data / PHI.
Practical Example

A list of internal team birthdays is 'Internal'. A patient's medical history is 'Confidential/Restricted'.

15 Data Retention

  • Delete or archive data per policy.
  • Don't keep logs or backups forever.
Follow the 'Clean Desk & Clean Drive' policy.
Practical Example

Set an S3 Lifecycle Policy to automatically move logs older than 1 year to Glacier or delete them.

16 Public Wi-Fi / VPN

  • Use the company VPN on untrusted networks.
  • Public Wi-Fi (Cafes/Airports) is high risk.
Practical Example

If working from Starbucks, connect to the Mindbowser VPN before accessing Epico or HROne.

17 Approved Comms

  • Official data stays on official tools.
  • No customer data on personal WhatsApp/Gmail.
Practical Example

Send project files via Epico or Google Drive links, never as attachments in a personal Telegram chat.

18 Incident: Phishing

  • Be cautious with links & attachments.
  • Check the sender's actual email address.
Don't click: "Urgent: Your password expired."
Practical Example

Hover over a link to see the real URL. If it says 'google-login.ru' instead of 'google.com', do not click.

19 Reporting Incidents

  • Use the official security mailbox/hotline.
  • Report immediately; time is critical.
Practical Example

Sent an email with sensitive data to the wrong person? Don't delete it; inform IT/Security so they can track the breach.

20 AI: Approved Tools

  • Use only company-vetted AI solutions.
  • Treat AI output as a draft, never as final.
Check the internal AI Usage Policy.
Practical Example

Use ChatGPT for brainstorming a function name, but never for processing a real list of patient names.

21 AI: Data Masking

  • Replace real data with generic placeholders.
  • Describe logic abstractly to the AI.
Practical Example

Instead of: "Find the error in this patient record for Jane Doe...", Use: "Find the logic error in this JSON object with a string field..."

Stay Compliant.

  • Security is a continuous habit.
  • Your vigilance protects our customers.
  • When in doubt: Ask before you act.

Build Fast. Stay Safe.
